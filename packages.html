Libs Python para adaptar para o ecossistema JS/TS

A ideia aqui não é portar biblioteca Python por nostalgia. O recorte bom é outro:

Quais libs fazem uma empresa JS-first criar scripts, workers ou microserviços em Python porque o ecossistema JS não resolve aquela dor tão bem?

Esse é o tipo de projeto que pode virar portfólio forte, porque mostra uma dor real: reduzir dependência de Python em times que já trabalham com Node, TypeScript, browser, edge, Bun, Deno ou automações JS.

O filtro principal é simples: portar lib só fica interessante quando existe gap real, DX melhor e empacotamento sério. Não é sobre copiar uma lib famosa. É sobre transformar uma dor comum em uma solução mais acessível para o ecossistema JS.

Critério de escolha

Uma lib candidata precisa responder “sim” para pelo menos 3 perguntas:

  • Uma empresa JS-first criaria um script Python só para resolver isso?
  • O equivalente JS atual é fraco, fragmentado, abandonado ou chato de integrar?
  • Dá para entregar uma versão menor, bem documentada e TypeScript-first?
  • Funciona em Node e, quando possível, browser/edge?
  • Tem demo fácil para README, artigo e landing page?
  • Resolve uma dor que aparece em SaaS, automação, CRM, dados, documentos ou tooling?

Tier S — maior oportunidade real

CleverCSV → csv-doctor / smart-csv-kit

CleverCSV é uma lib Python focada em lidar com CSVs bagunçados. Ela tenta detectar melhor o “dialeto” do arquivo, como separador, aspas e estrutura, especialmente em arquivos exportados por sistemas antigos, ERPs, CRMs ou planilhas mal formatadas.

Essa é uma das melhores apostas porque CSV zoado é uma dor real de empresa: ERP antigo, exportação de CRM, base de lead, planilha de cliente, relatório financeiro, CSV com ponto e vírgula, vírgula, encoding quebrado, aspas inconsistentes e header feio.

Produto JS possível

const result = await parseMessyCsv(file, {
  detectDialect: true,
  normalizeHeaders: true,
  inferTypes: true,
  reportProblems: true
})

Por que faz sentido

  • Dor comum em SaaS B2B.
  • Demo fácil: “jogue seu CSV quebrado e veja virar JSON limpo”.
  • Escopo controlável.
  • Pode funcionar em Node e browser.
  • Serve para importação de leads, produtos, clientes, pedidos e relatórios.

Nomes possíveis

  • csv-doctor
  • messy-csv
  • smart-csv-kit
  • tabular-cleaner

Frictionless → table-contract / csv-schema

Frictionless é um framework Python para descrever, extrair, validar e transformar dados tabulares. A oportunidade aqui não é clonar tudo. A oportunidade é criar uma versão TS-first focada em upload de CSV e Excel em apps web.

Produto JS possível

const schema = defineTable({
  email: string().email(),
  company: string().min(2),
  revenue: number().optional(),
  country: enumOf(["BR", "US", "MX"])
})

const report = await validateTable(csvFile, schema)

Por que faz sentido

  • Toda empresa que importa planilha precisa validar coluna, tipo, valor obrigatório e erro por linha.
  • É uma dor muito comum em dashboards internos e SaaS.
  • Dá para gerar relatório visual de erros.
  • Dá para integrar com Zod, Valibot ou uma DSL própria.
  • Pode virar uma lib muito boa para front + back.

Nomes possíveis

  • table-contract
  • csv-schema
  • tabular-guard
  • sheet-guard

MarkItDown → file-to-markdown / any-to-md

MarkItDown, da Microsoft, é uma ferramenta Python leve para converter arquivos variados em Markdown, especialmente para uso com LLMs, indexing e pipelines de análise de texto.

Essa é uma oportunidade forte porque muita empresa JS-first está mexendo com LLM/RAG e precisa transformar PDF, DOCX, PPTX, XLSX, HTML e outros arquivos em texto usável.

Produto JS possível

const markdown = await convertToMarkdown(file, {
  preserveTables: true,
  includeMetadata: true,
  strategy: "llm-friendly"
})

Por que faz sentido

  • Muito atual por causa de LLMs/RAG.
  • Empresas Node acabam usando worker Python só para ingestão de documentos.
  • Pode começar pequeno: HTML, TXT, CSV, DOCX.
  • Depois evoluir para PDF, PPTX, XLSX e imagens.
  • Tem narrativa forte: “document ingestion for TypeScript apps”.

Nomes possíveis

  • file-to-markdown
  • any-to-md
  • md-ingest
  • doc-to-md

Risco: formatos de arquivo viram buraco negro. O ideal é começar com poucos formatos e deixar a arquitetura plugável.

Trafilatura → maintext-js / cleanpage

Trafilatura é uma lib e CLI Python para coletar texto da web. Ela inclui crawling, download, scraping, extração de texto principal, metadata e comentários, com saída para formatos comuns.

Essa lib entra muito bem no mundo de SEO, prospecção, scraping e RAG. Em JS existem parsers de HTML, mas a dor real é: “me dá só o conteúdo principal da página, sem menu, header, footer, ads e lixo”.

Produto JS possível

const article = extractMainContent(html, {
  includeMetadata: true,
  includeLinks: false,
  output: "markdown"
})

Por que faz sentido

  • Útil para SEO, scraping, monitoramento de concorrentes e RAG.
  • Pode rodar em Node.
  • Tem demo simples: URL → texto limpo.
  • Conversa com automações, n8n, crawlers e análise de sites.
  • Pode ser menor e mais prática do que uma stack completa de scraping.

Nomes possíveis

  • maintext-js
  • cleanpage
  • article-extract
  • readable-html

dedupe → entity-match / lead-dedupe

dedupe é uma lib Python para deduplicação, fuzzy matching e entity resolution em dados estruturados. Ela é útil para cenários como remover duplicados em planilhas, juntar listas de clientes e encontrar registros que representam a mesma entidade mesmo sem ID único.

Essa talvez seja uma das ideias mais próximas de prospecção, CRM, Snov.io, leads e bases comerciais.

Produto JS possível

const clusters = await dedupeRecords(records, {
  fields: ["company", "email", "phone", "address"],
  threshold: 0.87
})

Por que faz sentido

  • CRM sujo é uma dor universal.
  • Base de lead duplicada é muito comum.
  • Dá para começar sem ML pesado, usando scoring por regras + similaridade.
  • Pode evoluir para active learning depois.
  • Tem uso real em vendas, marketing, logística, financeiro e atendimento.

Nomes possíveis

  • entity-match
  • lead-dedupe
  • record-linker
  • fuzzy-records

Tier A — muito boas, mas com escopo mais perigoso

Pandera → dataframe-guard / table-validate

Pandera é uma lib de validação para objetos tipo dataframe. Para JS, a melhor abordagem não seria clonar dataframe. Seria criar uma lib para validar arrays de objetos, CSVs e JSON tabular.

Produto JS possível

const customers = tableSchema({
  id: column.string(),
  age: column.number().between(18, 120),
  email: column.string().email(),
  country: column.enum(["BR", "US", "MX"])
})

customers.validate(rows)

Por que faz sentido

  • Bom para pipelines Node.
  • Bom para importação de dados.
  • Bom para validar resposta de API externa em lote.
  • Pode funcionar junto com csv-doctor.

Risco: se ficar genérico demais, vira só mais um validador. Precisa focar em dados tabulares.

unstructured → doc-elements / document-partition

unstructured oferece componentes para ingestão e pré-processamento de documentos como PDFs, HTML, Word docs e outros formatos, com foco em fluxos de dados para LLMs.

Essa é uma oportunidade forte, mas grande. Melhor fazer uma versão pequena e plugável.

Produto JS possível

const doc = await partitionDocument(file)

doc.elements
// [
//   { type: "title", text: "..." },
//   { type: "paragraph", text: "..." },
//   { type: "list_item", text: "..." }
// ]

Por que faz sentido

  • Muito útil para RAG.
  • JS tem parsers isolados, mas falta uma API única boa.
  • Pode integrar com Markdown, embeddings e chunking.
  • Dá para começar com HTML/Markdown/TXT e depois ir para DOCX/PDF.

Risco: tentar suportar todos os formatos logo no começo.

pdfplumber → pdf-inspector / pdf-table-lite

pdfplumber extrai informações detalhadas de PDFs, incluindo caracteres, linhas, texto, tabelas e debugging visual. Essa é uma dor muito real, mas PDF é um inferno técnico.

O ideal não seria começar por “PDF parser geral”. Melhor limitar o escopo para PDFs text-based ou gerados por máquina.

Produto JS possível

const pdf = await inspectPdf(file)

const text = pdf.pages[0].text()
const tables = pdf.pages[0].tables()

Por que faz sentido

  • Empresas vivem extraindo dados de PDFs.
  • Muito comum em relatórios, boletos, propostas, catálogos e documentos financeiros.
  • Dá portfólio forte.
  • Pode ter uma demo visual muito boa.

Risco: PDFs escaneados exigem OCR. Melhor declarar suporte inicial apenas para PDFs text-based/machine-generated.

Camelot → pdf-table-extract

Camelot é uma lib Python focada em extrair tabelas de PDFs. Essa opção talvez seja melhor do que tentar portar algo tipo pdfplumber inteiro. Uma lib só para tabela em PDF text-based já resolve uma dor grande.

Produto JS possível

const tables = await extractPdfTables(file, {
  pages: "1-3",
  strategy: "lattice"
})

Por que faz sentido

  • Caso de uso claro.
  • Escopo mais fechado.
  • Empresas fazem muito isso em Python.
  • Pode exportar para CSV, JSON e Markdown.

Risco: ainda assim é PDF. Precisa ser honesto no README sobre limitações.

libpostal / pylibpostal → address-br / address-latam

libpostal é uma lib em C para parsing e normalização de endereços no mundo todo. Existem bindings para outras linguagens, mas normalmente dependem da lib C por baixo.

Por isso, eu não faria um clone mundial. Eu faria uma lib menor, focada em Brasil/LatAm.

Produto JS possível

parseAddressBR("Av Paulista, 1000 cj 32 - Bela Vista, São Paulo - SP")

// {
//   street: "Av Paulista",
//   number: "1000",
//   complement: "cj 32",
//   district: "Bela Vista",
//   city: "São Paulo",
//   state: "SP"
// }

Por que faz sentido

  • Endereço é dado sujo.
  • Muito útil para CRM, logística, marketplace e cadastros.
  • Uma versão BR/LatAm é mais realista do que competir com libpostal.
  • Pode incluir CEP, UF, cidade, bairro, número e complemento.

Risco: parsing de endereço sem base de dados geográfica tem limite. Precisa ser vendido como parser/normalizador, não como geocoding.

Tier B — boas libs de dev/tooling, mas menos “empresa criou Python por isso”

DeepDiff → audit-diff / deep-change

DeepDiff compara profundamente objetos Python e também inclui recursos como busca, hash, delta e extração. Em JS já existem libs de deep diff, mas muitas são ruins para auditoria humana, payloads grandes ou relatórios bonitos.

Produto JS possível

const diff = deepDiff(before, after, {
  ignoreOrder: true,
  pathFormat: "json-pointer",
  humanReadable: true
})

Por que faz sentido

  • Audit logs.
  • Admin panels.
  • Comparação de payloads.
  • Snapshots de API.
  • CMS e SaaS multi-tenant.

Onde encaixa bem: no Linq, por exemplo, para registrar mudança de configuração, billing, organização, links e QR codes.

Risco: concorrência grande. Tem que ser muito bom em DX, output e TypeScript.

glom → json-transform-spec

glom é uma lib Python para acessar e transformar dados aninhados. JS mexe com JSON o tempo todo, então a dor existe. O problema é que também existem várias libs parecidas.

Produto JS possível

const output = transform(input, {
  name: "user.profile.name",
  city: "company.address.city",
  firstOrder: "orders.0.id"
})

Por que faz sentido

  • Bom para integrar APIs externas.
  • Bom para normalizar payloads.
  • Bom para ETL pequeno.
  • Bom para automações.

Risco: precisa ter uma API muito melhor que lodash.get, object-path, JSONata e similares. Senão vira mais uma lib de transformação.

Não colocaria como prioridade agora

Great Expectations

É muito forte no mundo de data quality, mas grande demais para virar um port direto. A ideia boa aqui seria pegar apenas um pedaço: validação de CSV/JSON tabular para apps Node.

pdfplumber completo

Muito forte tecnicamente, mas escopo grande e cheio de caso edge. Melhor começar com pdf-table-extract limitado a PDFs text-based.

unstructured completo

Excelente inspiração, mas tentar suportar PDF, DOCX, imagem, OCR, HTML, chunking e metadata de uma vez vai virar um produto gigante.

Ordem recomendada de implementação

1. csv-doctor

Melhor equilíbrio entre dor real, escopo possível, demo fácil e utilidade para empresa.

MVP

  • Detectar separador.
  • Detectar encoding básico.
  • Normalizar headers.
  • Inferir tipos.
  • Gerar relatório de erros.
  • Exportar JSON.
  • Rodar em Node e browser.

Demo ideal

Uma página onde o usuário arrasta um CSV bagunçado e vê:

  • dialeto detectado;
  • headers normalizados;
  • prévia da tabela;
  • erros por linha;
  • JSON final.

2. entity-match

Muito útil para CRM, leads, Snov.io, bases comerciais e dedupe.

MVP

  • Similaridade entre strings.
  • Normalização de empresa.
  • Dedupe por campos configuráveis.
  • Clusters de registros parecidos.
  • Score explicável.

Demo ideal

Upload de uma lista de empresas duplicadas e resultado agrupado:

  • “Huawei Technologies”
  • “Huawei Brasil”
  • “Huwei”
  • “Huawei do Brasil Ltda”

3. file-to-markdown

Mais ambicioso, mas com muito apelo por causa de LLM/RAG.

MVP

  • TXT para Markdown.
  • HTML para Markdown.
  • CSV para tabela Markdown.
  • DOCX para Markdown.
  • Metadados básicos.

Depois

  • PDF text-based.
  • PPTX.
  • XLSX.
  • Chunking para LLM.
  • Estratégias de output: clean, compact, llm-friendly.

Checklist de empacotamento sério

Para qualquer uma dessas libs virar portfólio de verdade:

  • TypeScript nativo.
  • ESM-first.
  • CJS apenas se fizer sentido.
  • Tree-shaking funcional.
  • Zero dependências de runtime, ou dependências muito bem justificadas.
  • Funciona em Node + browser quando possível.
  • Compatível com Bun e Deno quando não exigir APIs específicas do Node.
  • CDN-ready via jsDelivr, esm.sh ou unpkg.
  • Testes com Vitest.
  • Fixtures reais e casos quebrados.
  • Benchmarks contra alternativas.
  • README com exemplos simples.
  • Docs com casos reais.
  • Playground online.
  • GitHub Actions.
  • Semver disciplinado.
  • CHANGELOG mantido.
  • Issue templates.
  • Contributing guide.

Aposta final

Se fosse para escolher uma única ideia para começar, eu iria de:

csv-doctor

É a melhor combinação de:

  • dor real;
  • escopo controlável;
  • utilidade empresarial;
  • demo visual;
  • potencial de artigo;
  • NPM/CDN;
  • uso em browser e Node.

Como segunda aposta, eu faria:

entity-match

Porque conversa diretamente com CRM, prospecção, base comercial, deduplicação e fuzzy matching.

E como terceira, mais ambiciosa:

file-to-markdown

Porque o mercado de LLM/RAG está criando muita demanda para ingestão de documentos.

A tese final seria:

Criar libs TypeScript para resolver problemas que hoje empurram times JS para scripts Python.

Isso é bem mais forte do que “portar libs Python para JS”.
